React Hooks 完整教程

发表于 2026-07-16 分类于 前端开发

useState

用来在函数式组件中定义状态。

示例：

useReducer

可以替代 useState 创建状态，适合管理大量状态或复杂状态逻辑。

useContext

接收祖先组件通过 Context.Provider 传递的数据。

useRef

创建 ref 属性，用来标记 DOM 元素或组件实例。

htmlFor 代替 for 属性，用来和 input 绑定

useImperativeHandle

配合 forwardRef 使用，自定义暴露给父组件的内容。

useEffect

替代类组件的 componentDidMount 和 componentDidUpdate。

  • 第一个参数：回调函数，挂载和更新时执行
  • 第二个参数：依赖数组，只有依赖改变才执行
  • **空数组 []**：只在挂载时执行一次

常见用法：

useLayoutEffect

与 componentDidMount、componentDidUpdate 调用阶段相同，同步执行。

  • useEffect — 异步执行
  • useLayoutEffect — 同步执行，在 useEffect 之前调用

推荐先用 useEffect，出问题再考虑 useLayoutEffect

useInsertionEffect

也是同步函数，执行顺序在 useLayoutEffect 和 useEffect 之前。

主要用于 CSS-in-JS 库注入样式。

useCallback

用来缓存函数，避免子组件不必要的重新渲染。

使用场景： 子组件使用了父组件的方法，父组件其他状态变化导致重新渲染时，子组件也会被重新渲染。用 useCallback 包裹函数可以缓存函数引用。

useMemo

用来缓存计算结果，针对状态优化性能。

使用场景： 子组件接收的数据是由多个属性组合而成的对象（如 const info = { name, age }），父组件重新渲染会导致对象重新创建。用 useMemo 包裹可以避免不必要的重新计算。

useCallback vs useMemo

Hook 缓存内容 返回值
useCallback 函数 函数
useMemo 计算结果 值

useCallback(fn, deps) 相当于 useMemo(() => fn, deps)

useTransition

定义修改数据时的非阻塞机制。

  • isPending — 布尔值，是否处于等待状态
  • startTransition — 开启过渡的方法

用于将更新标记为非紧急更新，不阻塞 UI 响应。

useDeferredValue

传递一个值，返回该值的延迟副本。

用于延迟更新非紧急部分的 UI，类似防抖效果。

useDebugValue

用来标记自定义 Hook，可以在 React 开发工具中查看。

useId

生成唯一 ID，可用于无障碍属性等。

不能作为 CSS 选择器，也不能用来获取 DOM

useSyncExternalStore

允许组件订阅外部存储（如 Redux 等状态管理库）。

Hooks 使用规则

  1. 只在最顶层使用 Hook — 不要在循环、条件或嵌套函数中调用
  2. 只在 React 函数中调用 — React 函数组件或自定义 Hook
  3. 自定义 Hook 以 use 开头

Redux 完整教程（含 RTK）

发表于 2026-07-16 分类于 前端开发

Redux 介绍

Redux 是 React 中全局状态管理插件

  • 中文官网：https://cn.redux.js.org/

React 只是 DOM 的抽象层，并不是完整的 Web 应用解决方案。它没有涉及：

  • 代码结构
  • 组件之间的通信

2013 年 Facebook 提出 Flux 架构思想，2015 年 Redux 出现，将 Flux 与函数式编程结合，成为最热门的前端架构。

如果你不知道是否需要 Redux，那就是不需要它。只有遇到 React 实在解决不了的问题，才需要 Redux。

适用场景

不需要 Redux：

  • 用户使用方式简单
  • 用户之间没有协作
  • 不需要与服务器大量交互
  • 视图层只从单一来源获取数据

需要 Redux：

  • 用户使用方式复杂
  • 不同身份用户有不同使用方式
  • 多个用户之间可以协作
  • 与服务器大量交互或使用 WebSocket
  • View 要从多个来源获取数据

组件层面判断：

  • 某个组件的状态需要共享
  • 某个状态需要在任何地方都可以拿到
  • 一个组件需要改变全局状态
  • 一个组件需要改变另一个组件的状态

设计思想

  1. Web 应用是一个状态机，视图与状态一一对应
  2. 所有状态保存在一个对象中（唯一数据源）

三大原则

  • Single Source of Truth — 唯一的数据源
  • State is read-only — 状态是只读的
  • Changes are made with pure function — 数据改变必须通过纯函数完成

容器组件 vs 展示组件

展示组件 容器组件
作用 描述如何展现（骨架、样式） 描述如何运行（数据获取、状态更新）
直接使用 Redux 否 是
数据来源 props 监听 Redux state
数据修改 从 props 调用回调函数 向 Redux 派发 actions
调用方式 手动 通常由 React Redux 生成

Immutable.js

JavaScript 数据修改问题

JS 对象和数组是引用类型，直接赋值会指向同一块内存地址，修改一个会影响另一个。

浅拷贝（Object.assign、展开运算符）只能拷贝第一层，深层数据仍然共享引用。深拷贝（如 lodash 的 cloneDeep）又非常消耗性能。

什么是不可变数据

不可变数据（Immutable Data）一旦创建就不能更改。任何修改或添加删除操作都会返回一个新的 Immutable 对象。

原理：持久化数据结构 + 结构共享

  • 旧数据保持可用且不变
  • 只修改变化的节点和受影响的父节点，其他节点共享

优缺点

优点：

  • 降低 mutable 带来的复杂度
  • 节省内存（结构共享）
  • 历史追溯性（时间旅行）
  • 拥抱函数式编程

缺点：

  • 需要重新学习 API
  • 资源包大小增加（源码约 5000 行）
  • 容易与原生对象混淆

使用步骤

  1. 安装：cnpm install immutable redux-immutable
  2. 模块中用 Map 包裹 state
  3. 使用 state.set() 修改状态
  4. 用 redux-immutable 的 combineReducers 合并
  5. 组件中用 getIn() 获取数据
